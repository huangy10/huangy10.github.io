LTE-V2X Mode 4 拥塞控制

  • KNOWLEDGE-BASE
  • ACADEMIC
  • PAPER-READING
  • ADEL2019FIRST
| Views:

文章的标题是A First Investigation of Congestion Control for LTE-V2X Mode 4，是2019年IEEE WONS的会议文章。

这篇文章讨论了C-V2X Mode 4的资源调度中存在的碰撞问题。并提出了一种拥堵控制算法(Congestion Control Mechanism)【~碰撞也可以被视为一种拥堵】。In this work, we focus on the interactions between the strategies used by the LTE-V2X Sidelink Mode 4 for autonomous resource allocations - LBT (Listen-before-Talk) and Semi-Persistent (SPS) - and decentralized congestion control (DCC) mechanism.

1 Intro部分的新内容

  • Mode 4 构建了类似 ITS-G5 的 ad hoc 系统，因此同样面临冲突问题
  • 拥塞控制是指调节通信的配置参数来减少冲突的方法，这里的通信参数包括
    • 传输速率（应该指通信频率，而非数据率）
    • 传输功率
    • 调制参数
    • offload to alternative channels
  • 文章关注DCC机制对于Mode 4的性能影响
  • 使用ns3进行仿真
  • 提出修改LTE-V2X Mode 4的调度机制以适应DCC的需要
  • 一般的DCC算法会更多地尝试调节传输频率，因为这个更加可控一些。这种方法称为了ETSI的官方DCC机制。
  • 文章宣称目前还没有文章研究DCC和Mode 4之间的互动。

2 关于LTE-V2X的新内容

  • 所有的子信道有相同的大小
  • 3GPP LTE 规范 36.213 给出的可能的子信道大小：4, 5, 6, 8, 9, 10, 12, 15, 16, 18, 20, 25, 30, 48, 50 RB；以及可能的子信道的数量：1, 3, 5, 10, 15 or 20。
  • 每个ITS Station可以使用一个或者多个子信道进行通信。
  • PSSCH占用的RB的数量取决于payload的大小、子信道划分方式，以及MCS
  • SCI 格式 1的内容(32bit):
    • Priority: 消息的优先级
    • Resource Reservation: 只在 Mode 4 中使用，宣布基于感知决策占用的资源的长度。取值定义见 图 4
    • Frequency resource location: 定义PSSCH使用的RB资源
    • Time Gap: the number of subframes gap between the first and the optional second PSSCH transmission
    • MCS
    • Retransmission Index: indicates if the PSSCH refers to the first or the optional second transmission
    • Reserved bits

  • 除了重选计数到0这个条件以外，当需要发送的消息的大小超过了当前的子信道能够容纳的上限以后，也需要重新申请资源。

    这里有一个问题，这个节点如何告知周围的节点自己需要因为资源的传输能力不够而重选资源呢？应该也是通过SCI消息吧？这里的机制还没有论述的很清楚。

  • 在LTE-V2X中，节点有一定的概率将同一个包发送两次以确保鲁棒性。如果重传启用了，那么节点需要根据SB-SPS流程重新选择一个资源用于传输重传包，重传资源的时间未知需要在原包的发送位置的前后15ms处。

    这里的「重传」并非是在传输失败以后尝试第二次，若启用了重传，那么每个包都会被传输两次。

3 DCC

在DCC中所有的节点需要协作以避免信道饱和，同时还需要保证节点接入的公平性。定义两个量 Channel Busy Ratio (CBR) 和 Channel Occupancy Ratio (CR)。

  • CBR: 定义为在资源池中，RSSI值超过了一定阈值的子信道的数量。这个检测范围是过去的100个子帧，也即100ms。CBR反应了信道状态。
  • CR: 在子帧 \(n\) 处计算，定义为在子帧 \([n - a, n- 1]\) 内用于传输的子信道的数量和在 \([n, n + b]\) 内将被用于传输的子帧的数量的总和，除以监听区间内子帧的总和。\(a\) 和 \(b\) 的取值限制为 \(a + b + 1 = 1000, a \geq 500\)。CR反应了发送者端的信道利用效率。

CBR的取值范围可以划分为16段。在每段CBR取值中，规定一个CR不应该超过的阈值。对于每个发送者，根据探测到的CBR值得到CR阈值，如果当前的CR超过了阈值，那么该节点需要降低自己的CR。这里降低CR的方式可以有如下几种：

  • Drop packet retransmission: 如果重传启用了，可以关闭（这篇文章不考虑这点，文章这里假设重传总是关闭的）。
  • Drop packet transmission: 直接丢到包，注意只是丢掉当前的包，后续预留的资源还是可以继续使用。
  • 调整MCS: 可以提高MCS，提高传输速率，从而减少对于子信道的占用。不过提高MCS会降低可靠性。（这篇文章不考虑这个方法，文章驾驶MCS固定为7）
  • 调整传输功率: 这可以降低区域内的CBR，从而事实上提高CR的门限。（这篇文章也不考虑，因为太复杂，而且只有在所有车辆都启用这一机制之后才有意义this technique is not considered in this study as it notoriously complicated to fine-up (it can lead to oscillations), and would make sense only if all the stations are forced to use this technique）

4.2 仿真结果

在上面的场景中，传输损耗是PDR下降的主要因素，而非通信碰撞。而在70km/h的场景下，车辆密度较高，我们已经能看到在较近的通信距离上PDR有明显的下降。

在仿真配置中，本文将信道带宽划分为3个子信道，选择窗口的大小是100ms，即总共可调度的资源块是300个。文章这里试图调查当有250个车辆需要传输时的网络性能。文章这里对比了启用和不启用DCC两种情况。结果见 图 12 。可见DCC可以有效地降低CBR的水平。但是在 图 13 中可看到DCC对于PDR有负面作用。

4.3 碰撞的问题

文章的讨论部分(VII)讨论到了SPS的碰撞问题。SPS机制下部碰撞则以，一旦两个车辆选择了同一个资源块，由于半双工的工作特点，发送者自己无法感知到碰撞，因此在接下来的很长一段时间内会不断地重复碰撞。

仿真结果中DCC会对PDR有负面影响。这个原因可能是LTE中的资源保留方式确定的。SCI消息中Reservation field会告知其他车辆发送者下一次使用其占用的子信道的时间，如果车辆接下来不会使用这个资源了，这个字段会被设置成0。注意SCI信息只描述了紧接下来的下一轮的发送。当DCC机制丢掉一个包时，对应的SCI信息也丢掉了，这是其他车辆会认为后续的资源被释放了，从而有可能被其他车辆选中，带来碰撞。上面的过程可以参考 图 14 。

另外，DCC的packet drop会导致资源浪费。因为释放出来的那个资源块并没有被其他节点利用起来。

5 结论

  • 基于 Packet drop 的 DCC 机制不适合 Mode 4，这是由于SPS机制的特点导致的
    • 可嫩的方案是的在丢包时设置 RC = 0，强制节点重选资源
