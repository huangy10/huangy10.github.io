2 关于C-V2X标准的一些关键内容

下面的内容以 Release 14 的内容为主。

2.1 物理层信息

2.1.1 频谱与调制

C-V2X由于是使用蜂窝网架构，所以很容易被误认为是和5G一样是使用的几十GHz的频段。实际情况并非如此。C-V2X通信仍然采用的是sub-6GHz频段。2018年工信部印发了《车联网（智能网联汽车）直连通信使用5905-5925MHz频段管理规定（暂行）》的通知。这个通知除了规定使用的频段以外，还规定了发射功率等其他参数，具体如下：

  • 发射功率限值(EIRP)
    • 车载或便携无线电设备: 26dBm
    • 路边无线电设备: 29dBm
  • 载频容限
    • \(\pm 0.1 \times 10^{-6}\)
  • 邻道抑制比
    • 大于31dB，按信道积分功率有效值检波方式测试

注意到工信部这里给出的频率范围和WAVE使用的频率范围是重叠的。另外这里只给出工信部的规定，尚不确定世界上其他国家和地区，尤其是美国和欧洲的标准部门是如何规定的，这里还需要补充

以上是工信部的标准。在研究论文中也提到了更广为接受的标准。基于Configuraiton of the C-V2X Mode 4 Sidelink PC5 Interface for Vehicular Communications中的内容，C-V2X的一般特点如下：

  • 支持10MHz或者20MHz的信道
  • 使用SF-FDMA (Single-Carrier Frequency-Division Multiple Access)【这应该算MAC层了？】

C-V2X最关键的特征是将20MHz的带宽，同时从时域和频域进行切割。将1ms长度 \(\times\) 180kHz带宽1的时、频屏幕上的方块称为一个资源块(Resource Block, RB)。其中，时间尺度上这样的1ms称为一个子帧(sub-frame，也被称为Transmission Time Interval (TTI))。同一个子帧上的一组RB（应该是连续的）称为子信道(subchannel)，子信道中包含的RB的数量一般取决于调制编码方式（MCS）和包的大小。如下图所示：

2.1.2 LTE-Uu与PC5接口

LTE-Uu是蜂窝网的上下行链路使用的接口，PC5是Sidelink使用的接口。

2.1.3 频谱共享问题(ITS-G5)

这一部分主要参考的是CAR2CAR通信协会的一篇长文。其中第5.2部分讲了ITS-G52和LTE-V2X并存的问题。这篇文章的结论是由于两种通信协议的信道接入方式完全不同（CSMA/CA using OFDM v.s. SC-FDMA)，因此让二者平等地共享频段是不可能的a coexistence based on equal priorities in the same band is not possible。强行让二者共处会严重损害基于CSMA/CA的ITS-G5的性能。

对于这问题，有两种解决方案：

  1. 更新3GPP的空中接口，使之与ITS-G5的标准兼容Update the 3GPP air-interface to be compatible with the ITS-G5 interface including a new access method or reuse the ITS-G5 approach.
  2. 让两个系统采用不同的频段错开。这里应该是3GPP方面去选择新的频段，毕竟ITS-G5标准更加成熟。Use a different frequency band for the operation of the two systems including a sufficiently large frequency separation.

频谱效率最高的方法自然是方法1。例如ITS-G5/WLAN并存方面的研究提出强制要求一个10MHz的物理层探测器，在ITS-G5 A/B/D波段式，要有能力检测ITS-G5 preambles。方法2是目前 5GAA3 提出来的，理论上能彻底解决问题，不过需要使用更多的频谱资源。

在3GPP Rel-15提出New Radio之后，C-V2X将会使用above-6GHz频段，且自Rel-15之后的标准不会再兼容Rel-14。从中这个角度来看，3GPP应该是采用了上面的方案2。

2.1.4 一些关键问题(FAQ)

  1. C-V2X的频段具体是多少，每个信道带宽是多少，有多少个信道；
    目前可以确定的是信道带宽是10MHz或者20MHz，尚未看到文献提到C-V2X有多个信道。具体的频段目前看到的文献大部分没有提到。只查到工信部的要求是5905-5925MHz。

    5G NR (New Radio)中提出让V2X使用高于6GHz的频段4，5G NR可以为未来发布的协议标准提供更高级的服务，但是Release 14中提出的C-V2X标准的会是基础安全通信的核心基础。5G NR不会照顾向前兼容的问题，更像是添加一个额外的通信接口和信道。因此在研究中将C-V2X理解为使用5.9GHz ITS频段没有问题。

  2. C-V2X的有效传输距离一般是多少；
    有文献提到C-V2X的有效传输距离一般高于WAVE5。另外，下面第四个问题中的 图 4 也给出了大概的通信距离。

    LTE-V2X的距离优势是C-V2X的物理层带来的，而非其MAC层设计的优势。有文章研究尝试了结合C-V2X的物理层和IEEE802.11p的MAC层，发现可以取得和C-V2X相当的PRR(Pacet Reception Ratio)性能，且在UD(Update Delay)指标上更好。【由于PRR性能是以距离为横轴的曲线，因此这里称PRR性能相当，意味着通信距离相当】

  3. C-V2X传输的数据率一般是多少；
    尚不清楚。 在这篇Survey里面提到了每个RB上的符号数，那么在根据编码信息得到每个符号代表的比特数，就能够得到数据率。

    关于符号数: 一个RB上一共十四个符号，分别是4个解调参考符号(Demodulation Reference Symbols, DMRS)；9个数据符号；一个空符号，用来作为保护间隔。

    注意，目前我们还没有看到有文章称C-V2X的数据传输速率相比IEEE802.11p有显著的优势。反而有文章提到C-V2X对于大小不固定的包的处理不好。

  4. 延时性能
    Mode 3 和 Mode 4 的延时限制一般是 20 到 100 毫秒之间6。

  5. 抗干扰能力：SINR门限一般会是什么水平；
    尚不清楚

  6. 是否支持双工。（WAVE支持一辆车带两套天线，一套收一套发，但是需要在不通的信道。WAVE总共提供了7个信道）
    尚不清楚。 应当不支持

  7. 仿真工具

    • LTEV2Vsim: 基于MATLAB，也允许仿真IEEE802.11p。
    • Performance Analysis of Sensing-Based Semi-Persistent Scheduling in C-V2X Networks: 这篇文章提了一个Mode 4的开源仿真器。
    • 👆上面的文章中提到 NIST 团队在 NS-3的基础上做了支持Mode 1和Mode 2的仿真器。上文作者的仿真器是基于 NIST 团队的工作而来。

  8. RB的大小(时域)
    RB的大小不同的文章有不同的说法，频域上都是统一的，带宽为180kHz，时域上有0.5ms和1ms两种说法：

    • 0.5ms:
      1. Performance Analysis of Sensing-Based Semi-Persistent Scheduling in C-V2X Networks
      2. A Comparison of the V2X Communication System: ITS-G5 and C-V2X
    • 1ms:
      1. Better Platooning Control Toward Autonomous Driving: An LTE Device-To-Device Communications Strategy That Meets Ultralow Latency Requirements
      2. Configuraiton of the C-V2X Mode 4 Sidelink PC5 Interface for Vehicular Communications
      3. A First Investigation of Congestion Control for LTE-V2X Mode 4

    注意有些文章尽管说RB的长度是0.5ms，但是调度单位仍然是1ms

2.2 MAC层帧结构

2.2.1 PSSCH与PSCCH

PSSCH的全称是Physical Sidelink Shared Channel, PSCCH

PSCCH部分参见PSCCH

2.2.2 Transport Block and SCI

SCI 指 Sidelink Control Information。在Release 14中引入了SCI格式1，SCI信息在PSCCH上传输，占用两个RB。在Mode 4中SCI信息需要和消息主体（即TB）同时传输。

SCI格式1负责通知接收者一些关键信息，包括：

  • RRI (Resource Reservation Interval)
  • 初始传输和重传的频谱位置frequency location of initial transmission and retransmission (up to one retransmission is supported in Mode 4)
  • 初始传输和重传之间的间隔\(SF_{gap}\)
  • 调制编码方式(MCS)
  • 其他信息

2.3 MAC层SB-SPS

下面的两个章节如果看不太懂，这篇论文里面的介绍会更加清晰一些。

2.3.1 半持久与重选计数

C-V2X在MAC使用的机制是基于感知的半持久调度(Sensing-Based Semi-Persistent Scheduling)。其中「半持久」的概念，体现在重选计数器(Reselection Counter, RC)这个设计上。每当资源（选定的资源的单位一般是子信道）选定（选定的方法我们后面会提到）时，在后面的若干次传输中会继续使用同一个资源。这个「若干次」即由重选计数表示。在资源选定时，同时设置RC的值，设置方法为从5到15内随机选出。之后每经过一次传输，RC减一。当RC为0时，则面临重选资源的问题。

在需要面临资源重选时，节点可以选择不进行重选，继续使用现有的资源。这一决策一般是随机的，即节点以概率\(P\)继续使用当前的资源，即以概率\(1 - P\)选择新资源。 一般\(P \in [0, 0.8]\)。有些文章（如C-V2X Assisted mmWave V2V Scheduling）没有提到这个重选概率的问题，而是总是在RC为0时进行资源重选。这是可以认为\(P = 0\)。

RC的值需要包含在SCI消息中。

2.3.2 基于感知

  1. 当需要选择资源时，车辆\(V\)可以在时间\(T\)内做出决策，这个\(T\)为最大的建立延时，一般是100ms或者更低。这个时间段被称为选择窗口(Selection Window)。在选择窗口内，\(V\)需要确定候选的单子帧资源(Candidate Single-Subframe Resources, CSR)。具体而言，CSR是同一个子帧上的一组相邻的子信道。

    选择窗口的作用：选择窗口实际上规定资源块在时间上的周期划分方式。例如选择窗口100ms，频域上划分成3个子信道，那么可调度的资源块一共就是300个。7

  2. 车辆\(V\)在选择窗口\(T\)之前监听1000个子帧内（称为感知窗口，Sensing Window）的所有包传输情况。车辆然后建立起一个包含了在感知窗口中所有的CSR的列表\(L_1\)，除非【这里的除非是指如果CSR满足了下面两个条件就不会加入\(L_i\)列表】：

    1. \(V\) 收到了来自其他车辆的SCI信息，且这一SCI信息表示该车辆将会使用\(V\)在接下来要使用的网络资源；
    2. \(V\) 从比别的节点使用的RB上检测出来的平均Reference Signal Received Power (RSRP)高于阈值；这里的RSRP阈值是可配置的参数。

    车辆在这两个条件需要同时满足时\(V\)才会将对应CSR排除在\(L_1\)之外。这个步骤需要确保\(L_1\)中至少包含了20%的资源总量，否则这一步会一直循环知道20%的目标达到。而在每轮循环中，RSRP阈值会升高3dB。

  3. \(V\)创建第二个CSR的列表\(L_2\)。\(L_2\)中的CSR总数必须占CSR总数的20%。\(L_2\)来自\(L_1\)中最优最低的平均RSSI的CSR。当所有车辆的传输频率是10Hz时，这里的平均RSSI的计算范围是过去的第\(T_{CSR} - 100 \times j, j \in N, 1 \leq j \leq 10\) 子帧，参考上图。车辆\(V\)随机从\(L_2\)中选择一个CSR使用，并在接下来的Reselection Counter次传输中保持使用这一资源。

2.3.3 问题(FAQ)

  1. SB-SPS是否存在冲突问题。
    我还没有看到有文献讨论SPS机制中是否存在冲突问题，甚至有文章将SPS称为无冲突协议。但是可以说任何分布式的资源调度方式不出现冲突是不可能的。一方面信息无法保证一致，因此可能两个临近车辆会选择同一个资源块。另一方面，由于车辆的运动原本无冲突的两个节点会彼此靠近而导致冲突。

    A First Investigation of Congestion Control for LTE-V2X Mode 4 这篇文章里面讨论了冲突的问题。

  2. SCI中的具体信息内容。
    提这一点是想关注SCI中是否会包含本节点周围节点的资源分配信息，这涉及到是否会有隐藏终端问题。
